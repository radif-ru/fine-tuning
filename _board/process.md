# Процесс выполнения задачи

Пошаговый алгоритм работы над одной задачей из спринта.

## Предварительные условия

- [ ] Прочитан `CLAUDE.md` в корне проекта
- [ ] Прочитан `_docs/instructions.md`
- [ ] Известен текущий спринт и задача
- [ ] Создана feature-ветка (если это не инфраструктурный спринт 00)

## Шаги выполнения

### 1. Подготовка (перед кодированием)

1. Прочитай полное описание задачи в файле спринта
2. Определи затрагиваемые файлы
3. Если задача меняет поведение системы — прочитай связанные документы в `_docs/`
4. Сформулируй план реализации (в уме или кратко в чате)
5. **Зафиксируй начало работы:**
   ```bash
   git add _board/sprints/NN-*.md
   git commit -m "chore(plan): start task NN.M.O"
   ```

### 2. Реализация

1. Если есть DoD (Definition of Done) — следуй ему
2. При добавлении фичи:
   - Напиши/обнови тест (красный)
   - Реализуй фичу (зелёный)
3. При исправлении бага:
   - Напиши тест, воспроизводящий баг (красный)
   - Исправь баг (зелёный)
4. Следуй `_docs/instructions.md` § «Стиль кода»
5. Никаких `print` — только `logging`
6. Type hints обязательны

### 3. Проверка (до коммита кода)

1. **Прогнать тесты:**
   ```bash
   pytest -q
   ```
   Все тесты должны быть зелёными.

2. **Проверить покрытие (опционально):**
   ```bash
   pytest --cov=app --cov-report=term-missing
   ```

3. **Проверить линтер:**
   ```bash
   ruff check app tests
   ruff format --check app tests
   ```

4. **Проверить типизацию (опционально):**
   ```bash
   mypy app
   ```

### 4. Обновление документации

Если задача меняет поведение системы:

1. Найди соответствующий документ в `_docs/`
2. Обнови описание в соответствии с реализацией
3. Обнови `_docs/current-state.md`
4. Коммит документации:
   ```bash
   git add _docs/
   git commit -m "docs(architecture): обновить описание X после задачи NN.M.O"
   ```

### 5. Финальный коммит

1. Добавь изменённые файлы:
   ```bash
   git add app/ tests/
   ```

2. Создай коммит по Conventional Commits:
   ```bash
   git commit -m "feat(models): добавить поддержку LoRA для Phi-3
   
   - Реализована конфигурация LoRA через PEFT
   - Добавлены тесты для нового функционала
   - Обновлена документация
   
   Closes: NN.M.O"
   ```

### 6. Завершение задачи

1. Обнови статус задачи в файле спринта на **Done**
2. Обнови сводную таблицу задач
3. **Зафиксируй завершение:**
   ```bash
   git add _board/sprints/NN-*.md
   git commit -m "chore(plan): complete task NN.M.O"
   ```

4. Если это последняя задача спринта:
   - Обнови `plan.md` — перенеси спринт в «Закрытые»
   - Обнови `_docs/current-state.md`
   - Создай финальный коммит спринта

## Правила работы со спринтами

1. **Один спринт = одна ветка**
   - Ветка: `feature/NN-short-name`
   - Не смешивай задачи разных спринтов в одной ветке

2. **Исключение: Спринт 00**
   - Инфраструктурный спринт ведётся на `main`
   - Промежуточные коммиты `chore(plan)` могут быть пропущены

3. **Коммиты задачи**
   - `chore(plan): start task NN.M.O` — начало
   - `feat(...)` / `fix(...)` / `test(...)` — реализация
   - `docs(...)` — обновление документации
   - `chore(plan): complete task NN.M.O` — завершение

4. **Pull Request**
   - После закрытия спринта — создай PR в `main`
   - Опиши изменения по шаблону
   - Убедись, что CI проходит (тесты зелёные)

## Чеклист перед коммитом

- [ ] Тесты проходят: `pytest -q` → зелёный
- [ ] Код соответствует стилю проекта
- [ ] Type hints присутствуют в публичных API
- [ ] Нет `print`, только `logging`
- [ ] Документация обновлена (если менялось поведение)
- [ ] Сообщение коммита по Conventional Commits
- [ ] Коммит атомарный (одна логическая единица)

## Быстрые команды

```bash
# Проверка перед коммитом
pytest -q && ruff check app tests

# Полная проверка
pytest --cov=app --cov-report=term-missing && ruff check app tests && mypy app

# Обновление статуса задачи в спринте
# 1. Отредактируй файл спринта
# 2. git add _board/sprints/NN-*.md
# 3. git commit -m "chore(plan): start/complete task NN.M.O"
```
