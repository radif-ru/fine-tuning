# Процесс выполнения спринтов и задач

Документ описывает правила работы со спринтами и задачами: от открытия спринта до закрытия задачи.

---

## 1. Легенды

### 1.1 Статусы спринтов

| Статус | Обозначение | Описание |
|--------|-------------|----------|
| 📋 Запланирован | Planned | Спринт запланирован, но не начат |
| 🏃 В работе | In Progress | Спринт активно ведётся |
| ⏸️ Приостановлен | Paused | Спринт временно остановлен |
| ✅ Закрыт | Closed | Спринт завершён |

### 1.2 Приоритеты задач

| Приоритет | Обозначение | Когда использовать |
|-----------|-------------|-------------------|
| critical | P0 | Блокер для других спринтов, критичный баг |
| high | P1 | Основной функционал, важные фичи |
| medium | P2 | Улучшения, оптимизации |
| low | P3 | Nice-to-have, документация |

### 1.3 Статусы задач

| Статус | Описание |
|--------|----------|
| ToDo | Задача ожидает выполнения |
| Progress | Задача в работе (только одна на всей доске) |
| Done | Задача выполнена |
| Blocked | Задача заблокирована (внешние зависимости) |

### 1.4 Объём задач

| Объём | Примерная длительность |
|:-----:|:------------------------|
| XS | < 1 часа |
| S | 1–4 часа |
| M | 4–8 часов |
| L | 1–2 дня |
| XL | > 2 дней (разбить на smaller) |

## 3. Открытие спринта

**Выполняется по запросу пользователя** (кроме случаев явного запроса открывать автоматически).

1. **Создать ветку** от `main`:
   ```bash
   git checkout -b feature/<NN>-<short-name>
   ```

2. **Создать файл** `_board/sprints/<NN>-<short-name>.md` по шаблону §2.1.

3. **Обновить** `_board/plan.md` — индекс спринтов:
   - Добавить спринт в таблицу «Активные»
   - Заполнить: #, Название, Ветка, Статус = 🏃 В работе

4. **Закоммитить**:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md _board/plan.md
   git commit -m "chore(plan): открыть спринт <NN> — <название>"
   ```
   > Формулировка **строго «открыть»**, не «начать» и не «создать».

> **Push и merge делает пользователь сам.**

## 4. Выполнение задачи

### 4.1 Прочитать документацию

Обязательно перед началом:

1. `CLAUDE.md` — поведенческие гайдлайны LLM-агента.
2. `_docs/README.md` — навигация по документации.
3. `_docs/current-state.md` — потенциальные подводные камни.
4. `_docs/instructions.md` — правила разработки (git, стиль, тесты).
5. `_board/sprints/<active>.md` — выбираемая задача.
6. Документы из поля «Связанные документы» задачи.

При расхождении документа и кода — приоритет у кода.

### 4.2 Выбрать задачу

1. Открыть файл активного спринта `_board/sprints/<NN>-<short-name>.md`.
2. Найти **первую сверху** задачу со статусом `ToDo`.
3. Проверить «Зависит от»: все зависимости должны быть `Done`.
4. Если требуются внешние действия — пометить `Blocked`.

### 4.3 Начать задачу (Progress)

1. В файле спринта изменить статус задачи: `ToDo` → `Progress`.
2. Обновить «Сводную таблицу задач спринта».
3. Закоммитить:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md
   git commit -m "chore(plan): начать задачу <NN>.<stage>.<task>"
   ```

> В статусе `Progress` — **ровно одна** задача на всей доске.

### 4.4 Реализовать

Принципы из `_docs/instructions.md`:

- **Минимальный диф**, решающий проблему
- **Не править соседний код** без явной задачи
- **Не добавлять «гибкости на будущее»**
- **Соблюдать существующий стиль**

Цикл: правка → smoke-test → проверка.

**Для задач, меняющих `app/` — обязательно добавить/обновить тесты**.

### 4.5 Проверить (smoke-test)

```bash
# 1) Импорт не сломан
python -c "import app; print('OK')"

# 2) CLI доступен
python -m app --help

# 3) Тесты проходят
pytest -q
```

### 4.6 Закрыть задачу (Done)

1. Проверить Definition of Done по всем пунктам (отметить чекбоксы `[x]`).
2. **Обновить документацию** `_docs/` если меняется поведение.
3. В файле спринта изменить статус задачи: `Progress` → `Done`.
4. **Важно:** Обновить статус задачи **везде** где она упоминается:
   - В заголовке задачи (поле **Статус:**)
   - В «Сводной таблице задач спринта»
   - В Definition of Done (отметить выполненные пункты `[x]`)
5. Дописать в «Историю» спринта дату и коммит.
6. Закоммитить:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md
   git commit -m "chore(plan): закрыть задачу <NN>.<stage>.<task>"
   ```

## 5. Закрытие спринта

**Решение о закрытии принимает пользователь** (кроме случаев явного запроса закрывать автоматически).

Когда все задачи спринта выполнены (`Done`):

1. **Обновить файл спринта** `_board/sprints/<NN>-<short-name>.md`:
   - Поле **Закрыт:** YYYY-MM-DD
   - Поле **Статус:** ✅ Закрыт
   - Раздел **История:** строка о закрытии

2. **Обновить** `_board/plan.md` — индекс спринтов:
   - Перенести спринт из «Активные» в «Закрытые»
   - Заполнить колонки: Открыт, Закрыт, Статус

3. **Закоммитить**:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md _board/plan.md
   git commit -m "chore(plan): закрыть спринт <NN> — <название>"
   ```

> **Merge в `main` делает пользователь сам.**

---

## 6. Обновление документации

**Обязательно для задач, меняющих поведение.**

1. Найти соответствующий документ в `_docs/`.
2. Внести правки.
3. Обновить `_docs/README.md` § «Навигация» при необходимости.
4. Коммит `docs(<scope>): ...` — отдельно от кода.
