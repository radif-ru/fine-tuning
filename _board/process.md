# Процесс выполнения одной задачи

Документ описывает пошаговый алгоритм для закрытия **одной** задачи из активного спринта. Алгоритм универсален — не зависит от конкретной задачи. Содержание задачи, её Definition of Done и связанные документы берутся из `_board/sprints/<active>.md` и `_docs/`.

## 0. Перед началом сессии

Один раз на сессию работы:

1. `git status` пуст, ветка актуальна (`git pull`).
2. Активировано виртуальное окружение: `source .venv/bin/activate`.
3. Зависимости актуальны: `pip install -r requirements.txt` (если был добавлен/обновлён модуль).
4. **На активной ветке спринта** (см. `_board/plan.md` § «Индекс спринтов» → колонка «Ветка»).

### 0.1 Открытие нового спринта (если нужно создать)

При создании нового спринта (по запросу пользователя):

1. **Создать ветку** от `main`: `git checkout -b feature/<NN>-<short-name>`.
2. **Создать файл** `_board/sprints/<NN>-<short-name>.md` по шаблону из `plan.md`.
3. **Обновить** `_board/plan.md` — добавить спринт в «Индекс спринтов» (Active).
4. **Закоммитить** с сообщением: `chore(plan): открыть спринт <NN> — <название>`.
   - Формулировка **строго «открыть»**, не «начать» и не «создать».

> **Push и merge делает пользователь сам.** Пушить ветку или мержить в `main` — только по явному запросу пользователя.

### 0.2 Закрытие спринта

**Решение о закрытии спринта принимает пользователь.** Когда все задачи спринта выполнены (все `Done`):

1. **По умолчанию** — спросить пользователя: «Все задачи спринта выполнены. Закрыть спринт <NN>?»
2. **При явном запросе** пользователя (например, «реализуй сразу несколько спринтов подряд») — можно закрывать автоматически без лишних вопросов, переключаться на новую ветку и продолжать.
3. **Формулировка коммита**: `chore(plan): закрыть спринт <NN> — <название>`.
4. **Действия при закрытии**:
   - Обновить `_board/sprints/<NN>-<short-name>.md` — дата закрытия, статус `Closed`.
   - Обновить `_board/plan.md` — перенести спринт в «Закрытые».
   - **Не добавлять в коммит закрытия задачи из следующих спринтов** — только текущий спринт.

> **Merge в `main` делает пользователь сам** — только по явному запросу.

## 1. Прочитать документацию

**Обязательный шаг перед началом любой задачи.**

1. `CLAUDE.md` — поведенческие гайдлайны LLM-агента (think before coding, simplicity, surgical changes, goal-driven). Применяются к любой задаче.
2. `_docs/README.md` — навигация по проектной документации.
3. `_docs/current-state.md` — есть ли уже запись про затрагиваемую область (потенциальные подводные камни, легаси).
4. `_docs/instructions.md` — правила разработки (git, стиль, async, ошибки, секреты, тесты, документация). Перечитывать до автоматизма.
5. `_board/plan.md` — индекс спринтов: какой спринт активен и какая у него ветка.
6. `_board/sprints/<active>.md` — выбираемая задача и её соседи (зависимости).
7. Документы из поля «Связанные документы» выбранной задачи (например, `_docs/architecture.md` для задач по структуре; `_docs/training.md` — для тренировки).

При расхождении документа и кода — приоритет у кода. Документ обновится после задачи отдельным коммитом.

## 2. Выбрать задачу

1. Открыть `_board/plan.md` → найти активный спринт → открыть его файл `_board/sprints/<NN>-<short-name>.md`.
2. Найти **первую сверху** задачу со статусом `ToDo` (по этапам: сначала задачи этапа 1, затем этапа 2, …).
3. Проверить поле «Зависит от»: все указанные задачи должны быть `Done`. Если нет — взять предыдущую (по зависимостям, не по порядку).
4. Если требуются внешние действия (ТЗ, доступы, решения от пользователя), которых нет — пометить задачу `Blocked` с причиной, остановиться.

## 3. Перевести задачу в статус `Progress`

1. В файле спринта `_board/sprints/<NN>-<short-name>.md` у выбранной задачи изменить `**Статус:** ToDo` → `**Статус:** Progress`.
2. Обновить «Сводную таблицу задач спринта» в этом же файле (колонка «Статус»).
3. Закоммитить:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md
   git commit -m "chore(plan): начать задачу <NN>.<stage>.<task>"
   # пример: chore(plan): начать задачу 01.2.4
   ```

> Правило: в статусе `Progress` одновременно — **ровно одна** задача на всей доске.

### Порядок выполнения задач

Задачи выполняются по порядку номеров внутри этапа (1.1 → 1.2 → 1.3). **Исключения:**

- **Критичный баг** — можно выполнять вне очереди, но указать причину в поле «Зависит от» задачи (например, «Баг блокирует тестирование»).
- **Зависимость от другой задачи** — если задача A зависит от задачи B (указано в «Зависит от»), но B ещё не Done, можно:
  - Подождать выполнения B;
  - Или явно запросить разрешение пользователя на изменение порядка.

## 4. Подготовить контекст для реализации

1. Прочитать «Описание» и «Definition of Done» задачи целиком.
2. Открыть в редакторе файлы из секции «Затрагиваемые файлы» (если указана) или предполагаемые по описанию.
3. Если задача связана с конкретной подсистемой — открыть параллельно соответствующий документ из `_docs/`:
   - конфиг / логирование → `architecture.md` §3.2–3.3;
   - модели / LoRA → `architecture.md` §2, `training.md`;
   - данные → `architecture.md` §2.3, `training.md` §3;
   - тренировка → `training.md` §4–6;
   - инференс → `inference.md`.
4. Сформулировать (для себя или для LLM-помощника) краткий план шагов.

## 5. Реализовать задачу

Следуем `_docs/instructions.md` (минимализм, хирургические правки, цель → проверка):

1. **Самый минимальный диф**, решающий проблему.
2. **Не править соседний код** без явной задачи на это.
3. **Не добавлять «гибкости на будущее»** — реализуй ровно то, что нужно.
4. **Соблюдай существующий стиль** (импорты, нейминг, структура файлов).
5. Если встретился несвязанный баг или мёртвый код — записать в `_docs/current-state.md` §2 по шаблону из §5 этого документа, **не править** в этой задаче.

Цикл «правка → проверка»:

1. Внести правку.
2. Прогнать smoke-проверки (см. §6).
3. Если что-то поломалось — откатить и переделать.
4. **Для задач, меняющих код в `app/` — обязательно добавить или обновить тесты на новое поведение до зелёного состояния** (см. §6.3). Чисто-документационные задачи от этого правила освобождены — в DoD таких задач ставится `n/a`.

## 6. Проверка изменений

### 6.1 Минимальный smoke-test (всегда)

```bash
# 1) Импорт пакета не сломан
python -c "import app; print('OK')"

# 2) Сборка приложения (точка входа) — если затронуты main / core
python -c "from app.core.config import Settings; print(Settings.__name__)"

# 3) CLI доступен
python -m app --help
```

### 6.2 Целевые проверки (когда применимо)

- **Конфигурация** — `python -c "from app.core.config import Settings; print(Settings().log_level)"`.
- **Core** изменён — `pytest tests/unit/core -q`.
- **Utils** изменены — `pytest tests/unit/utils -q`.
- **`.env` / секреты** — `git status` не показывает `.env`; `git log -p | grep HF_TOKEN=` возвращает только примеры из `.env.example`.

### 6.3 Тесты

```bash
pytest -q                                      # все тесты
pytest tests/<затронутый-модуль> -q -v         # детально по подсистеме
pytest --cov=app --cov-report=term-missing     # покрытие (если установлен pytest-cov)
```

## 7. Проверить Definition of Done

1. Открыть в файле спринта раздел «Definition of Done» выбранной задачи.
2. По каждому пункту — отметить «выполнено / не выполнено».
3. Если хотя бы один пункт не закрыт — вернуться к §5.
4. **Обязательное обновление документации.** Если задача меняет поведение системы — обновить соответствующий документ в `_docs/` (см. §10). Это делается ДО коммита кода.
5. Прогнать `git status` — никаких лишних файлов (`*.log`, `.env`, `__pycache__/`, `data/*.db`).

## 8. Зафиксировать результат в git

**Перед каждым коммитом задачи (кроме `chore(plan): начать/закрыть задачу ...`) — прогнать `pytest -q` и убедиться, что он зелёный.** Если тесты падают (даже не связанные с задачей) — сначала чинятся, и только потом коммит. См. `_docs/instructions.md` §7.2 «Обязательное правило (зелёный pytest перед коммитом)».

Коммиты — атомарные, conventional commits, в императиве. Язык — по общему правилу проекта (`_docs/instructions.md` §0):

```bash
pytest -q                              # обязательно: должно быть зелёным
git add <изменённые файлы>
git commit -m "feat(<scope>): <что сделано>"
# или: fix(...) / refactor(...) / docs(...) / test(...) / chore(...)
```

Примеры (типичные scope этого проекта):

- `feat(models): добавить загрузку модели из HuggingFace`
- `feat(lora): реализовать конфигурацию LoRA через PEFT`
- `feat(data): добавить загрузчик JSONL датасетов`
- `feat(training): интегрировать Trainer с LoRA`
- `fix(config): обработать отсутствие .env файла`
- `refactor(trainer): вынести колбэки в отдельный модуль`
- `docs(architecture): уточнить поток обучения в §5`
- `test(models): покрыть загрузку модели тестами`
- `chore(plan): закрыть задачу 01.2.4`

## 9. Перевести задачу в статус `Done`

1. В файле спринта изменить `**Статус:** Progress` → `**Статус:** Done`.
2. Обновить «Сводную таблицу задач спринта».
3. Дописать строку в § «История изменений спринта»: дата + краткое описание + ссылка на коммит/коммиты.
4. Если это последняя задача спринта — обновить также `_board/plan.md` § «Индекс спринтов» (перенести спринт из «Активный» в «Закрытые», заполнить дату закрытия) и сводную таблицу в `_board/plan.md`.
5. Коммит:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md _board/plan.md
   git commit -m "chore(plan): закрыть задачу <NN>.<stage>.<task>"
   ```

## 10. Обновить документацию (обязательно для кодовых задач)

**Обязательный шаг для задач, меняющих поведение системы.** Выполняется ДО коммита кода (см. §7).

Если в ходе задачи изменилось поведение, отражённое в `_docs/`:

1. Найти соответствующий документ (`architecture.md`, `training.md`, `inference.md`, `configuration.md`, `testing.md`).
2. Внести правку.
3. Если изменилась структура каталога или появился новый документ — обновить `_docs/README.md` § «Навигация» и `_docs/project-structure.md`.
4. Если найден новый легаси-нюанс — добавить в `_docs/current-state.md` §2 (см. шаблон в §5 этого документа).
5. Коммит `docs(<scope>): ...` — отдельным коммитом перед кодовым коммитом.

## 11. Перейти к следующей задаче

1. Вернуться к §2 — выбрать следующую задачу `ToDo`.
2. Если все задачи спринта `Done`, а пунктов ToDo нет — спринт можно закрывать (см. `_board/plan.md` § «Правила работы со спринтами», п.4–5).
3. Если есть запрос пользователя на новый спринт — открыть новый `sprints/<NN+1>-<short-name>.md` (новая ветка `feature/<short-name>` от `main`).

## Короткая шпаргалка (TL;DR)

1. Прочитать `CLAUDE.md` → `_docs/README.md` → `_docs/current-state.md` → `_board/plan.md` → файл активного спринта → документы задачи.
2. Взять верхнюю `ToDo` задачу — проверить, что зависимости `Done`.
3. Сменить статус на `Progress` в файле спринта, коммит `chore(plan): начать задачу <NN>.<stage>.<task>`.
4. Минимальный диф → smoke-test → зелёный.
5. По каждому пункту DoD — отметка «выполнено».
6. Коммит правок (`feat/fix/refactor/docs/test/chore`).
7. Сменить статус на `Done`, коммит `chore(plan): закрыть задачу <NN>.<stage>.<task>`.
8. К следующей задаче.

## Чек-лист перед переходом к следующей задаче

- [ ] Definition of Done закрыт по всем пунктам.
- [ ] **Для задач, меняющих код в `app/` — добавлен или обновлён хотя бы один unit-тест на новое поведение** (обязательное правило, см. `_docs/instructions.md` §7.1). Для чисто-документационных задач — `n/a`.
- [ ] **`pytest -q` зелёный** (обязательное правило, см. `_docs/instructions.md` §7.2).
- [ ] Smoke-test пройден (импорт + ручная проверка CLI, если задача меняет UX).
- [ ] `git status` чист (нет лишних артефактов / `.env` / логов / `data/*.db`).
- [ ] **Документация обновлена там, где меняется поведение** (обязательное правило для кодовых задач, см. CLAUDE.md §0).
- [ ] Статус задачи в файле спринта = `Done`, сводная таблица актуальна.
- [ ] Если последняя задача спринта — `_board/plan.md` обновлён (спринт переведён в Closed).
