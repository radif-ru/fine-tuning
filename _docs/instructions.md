# Инструкции по разработке

## Общие принципы

Для поведенческих гайдлайнов LLM-агента — см. `CLAUDE.md` в корне проекта.

## 0. Язык проекта

**Единое правило: по умолчанию — русский, но не строго.** Если на другом языке смысл понятнее (имена символов, термины из кода, англоязычные технические выражения, служебные коммиты вроде `chore(plan): начать задачу ...`, цитируемые буквально фрагменты кода/логов) — используется он. Смешивать языки в одном сообщении/документе допустимо.

Применяется ко **всему без исключения**:
- Сообщения в чате с пользователем и внутренние рассуждения (chain-of-thought, scratchpad, планы шагов).
- Сообщения коммитов (см. `_board/process.md` §8).
- Комментарии в коде и докстринги.
- Тексты в `_docs/`, `_board/`, `README.md`, `.env.example`.
- **Логи** (сообщения `logger.info/warning/error/exception`, текст в лог-форматтере).
- Сообщения, которые бот отправляет пользователю (включая тексты ошибок и подсказок команд).

Технические идентификаторы (имена переменных/функций/классов/файлов, URL'ы, имена моделей, env-переменные, ключи словарей, литеральные значения протоколов) остаются латиницей — это часть кода, а не текста. Имена индустриальных терминов (LLM, LoRA, PEFT, RAG, fine-tuning) не переводим.

Если получил инструкцию или фрагмент на другом языке — переключаться целиком не нужно: переведи смысл и продолжай по общему правилу.

## 1. Git-дисциплина

- **Основная ветка** — `main`, рабочие ветки — `feature/<short-name>` (по одной на спринт)
- **Коммиты** — атомарные, в императиве, по [Conventional Commits](https://www.conventionalcommits.org/):
  - `feat(models): добавить поддержку Phi-3`
  - `fix(training): исправить утечку памяти`
  - `test(data): покрыть loader тестами`
  - `docs(config): обновить описание LoRA параметров`
  - `chore(plan): начать задачу 01.2.3`
- **Ритуал коммитов** — каждая задача из спринта проходит цикл `начать → feat/test/... → закрыть` тремя отдельными коммитами. Подробно — в `CLAUDE.md` §4 и `_board/process.md`
- `.gitignore` — обязательно содержит: `.env`, `venv/`, `.venv/`, `__pycache__/`, `logs/`, `data/`, `checkpoints/`, `outputs/`, `*.db`, `.pytest_cache/`
- **Секреты никогда не коммитить**

## 2. Стиль кода

- **Python** 3.10+, type hints **обязательны** в публичных функциях
- **Форматирование** — `ruff format`, длина строки 100
- **Импорты** — `ruff` / `isort`-совместимо: stdlib → third-party → local
- **Имена** — `snake_case` для функций/переменных, `PascalCase` для классов, `UPPER_CASE` для констант
- **Docstrings** — Google style, краткие
- **Никаких `print`** — только `logging`

## 3. Async-дисциплина

Проект синхронный (тренировка LLM не требует async), но при использовании I/O:
- HTTP запросы — `requests` или `httpx`
- Файловые операции — стандартные sync
- HuggingFace datasets — sync

## 4. Обработка ошибок

- Ожидаемые ошибки — ловить и превращать в понятные сообщения
- Неожиданные — логировать с `logger.exception(...)`
- Создавать иерархии исключений:

```python
class FineTuningError(Exception):
    """Базовый класс для ошибок проекта"""
    pass

class ModelLoadError(FineTuningError):
    """Ошибка загрузки модели"""
    pass

class DataLoadError(FineTuningError):
    """Ошибка загрузки данных"""
    pass

class TrainingError(FineTuningError):
    """Ошибка во время обучения"""
    pass
```

## 5. Логирование

- Конфигурация — `app/core/logging_config.py`
- Каждый этап обучения логируется
- Формат: `timestamp [LEVEL] [module] message | key1=value1 key2=value2`
- Ошибки — `logger.exception(...)` или `logger.error(..., exc_info=True)`
- Чувствительные данные — никогда в логи

Примеры:
```python
logger.info("Модель загружена | model=%s params=%d", model_name, num_params)
logger.info("Начало эпохи | epoch=%d/%d", current, total)
logger.warning("Мало данных | samples=%d min_recommended=%d", n, 100)
logger.error("Ошибка загрузки | path=%s", path, exc_info=True)
```

## 6. Секреты и конфиг

- Все секреты и настройки — из `.env` через `pydantic-settings`
- `.env` — в `.gitignore`, в репо — `.env.example` с описанием
- Не хардкодить пути, имена моделей, токены в коде

## 7. Тестирование

### Минимум

- `pytest` + `pytest-mock`
- Unit-тесты для всех компонентов
- Покрытие ≥ 70%

### Обязательное правило

Новый или изменённый код в `app/` **не принимается без unit-теста** на новое поведение.

### Запуск перед коммитом

```bash
pytest -q
```

Коммит **только при зелёном результате**.

## 8. Документация

- Язык — русский (см. §0)
- Технические идентификаторы — латиницей
- **Обязательное обновление** — перед коммитом задачи, меняющей поведение, обновить соответствующий документ в `_docs/`
- Перекрёстные ссылки — относительными путями

## 9. Процесс добавления фичи

1. **Описать поведение** — в соответствующем документе `_docs/`
2. **Написать тест** (красный)
3. **Реализовать** (зелёный)
4. **Отрефакторить**, прогнать линтер
5. **Обновить документацию**
6. **Прогнать `pytest -q`** — все зелёные
7. **Коммит**

## 10. Что НЕЛЬЗЯ делать

- Использовать `print` вместо `logging`
- Коммитить `.env` или реальные токены
- Писать код без type hints в публичных API
- Игнорировать тесты
- Удалять ранее существовавший мёртвый код без согласования
- Расширять scope задачи без явного согласования

## 11. Локальный запуск (dev)

```bash
# 1) окружение
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# 2) секреты
cp .env.example .env
# отредактировать .env — указать пути, модели

# 3) запуск тренировки
python -m app train

# 4) тесты
pytest -q
```
